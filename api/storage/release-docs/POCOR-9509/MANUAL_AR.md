
# OpenEMIS Alerts — Administrator Manual

## مقدمة إدارة OpenEMIS

يُعتبر هذا الدليل المرجع الموثوق لوحدة التنبيهات في OpenEMIS Core، التي تم تقديمها وتطويرها ضمن POCOR-9509. يغطي كل نوع تنبيه وكل خيار إعدادات وكل إجراء تشغيلي يحتاجه المسؤول. استخدم جدول المحتويات للانتقال مباشرة إلى القسم المتعلق بمهمتك. يتم توفير أمثلة عملية وأشجار قرار استكشاف الأخطاء في جميع أنحاء الدليل لحل المشاكل الشائعة بسرعة.

## 1. المقدمة {#1-introduction}

### 1.1 ما تفعله وحدة التنبيهات

تقوم وحدة تنبيهات OpenEMIS بإرسال إشعارات آلية إلى موظفي المدارس والوزارة عند ظهور حالات حرجة. يتم إرسال الإشعارات عبر البريد الإلكتروني أو الرسائل القصيرة أو كليهما، وفقًا لتكوين القاعدة. تشمل أمثلة الحالات المراقَبة تراكم أيام غياب الطالب فوق الحد المحدد، واقتراب تاريخ انتهاء عقد الموظف، واقتراب انتهاء صلاحية ترخيص مهني، واقتراب الموعد النهائي لطلب منحة دراسية، وتجاوز قضية غير محلولة لحد التصعيد. تعمل الوحدة باستمرار دون تدخل يدوي بعد تكوينها، وتحتفظ بسجل تدقيق دائم لكل رسالة مُرسلة.

### 1.2 من يجب أن يقرأ هذا الدليل

هذا الدليل موجّه لثلاث فئات. مسؤولو تقنية المعلومات في الوزارة الذين يقومون بنشر وصيانة OpenEMIS سيجدون أقسام البنية والتكوين واستكشاف الأخطاء هي الأكثر صلة. كبار مديري النظم المسؤولون عن سياسة الاتصالات سيستخدمون قسم مرجع أنواع التنبيهات وقسم تكوين القواعد. مهندسو النشر الذين يقومون بالتفعيل الأولي سيتبعون أقسام التكوين التشغيلي والاختبار. تستفيد الفئات الثلاث جميعًا من قراءة القسمين 2 و8 قبل التعامل مع مهام محددة.

### 1.3 ما هو الجديد في POCOR-9509

يُقدّم إصدار POCOR-9509 التغييرات التالية على البنية التحتية للتنبيهات:

- **خمسة أنواع تنبيه جديدة**: تصعيد القضية، صلاحية الترخيص، تجديد الترخيص، طلب المنحة الدراسية، وصرف المنحة الدراسية. يتم توثيق كل منها بالكامل في القسم 8.
- **شاشة قائمة انتظار التنبيهات**: شاشة جديدة ضمن Administration → Communications → Alert Queue توفر رؤية فورية لحالة خط الإرسال لكل رسالة معلقة أو مُرسلة أو فاشلة.
- **الحذف الجماعي لسجلات التنبيهات وقائمة الانتظار**: يمكن للمسؤولين الآن اختيار سجلات متعددة وحذفها بعملية واحدة، مما يتيح التنظيف السريع بعد أن تُنتج قاعدة مُعدّة بشكل خاطئ إدخالات غير مرغوبة.
- **وقت تشغيل مستند إلى Laravel**: تم نقل جميع معالجة التنبيهات من نصوص CakePHP Shell إلى أوامر artisan نظيفة تحت `api/app/Console/Commands/Alerts/`. يوفر ذلك معالجة أخطاء متسقة، وتتبع العمليات، وقابلية الاختبار.
- **التحكم في التدفق عبر `ALERTS_PROCESS_LIMIT`**: متغير بيئة واحد في `.env` يتحكم في الحد الأقصى لعدد الرسائل المعالَجة في كل تشغيل للجدولة، مما يمكّن المسؤولين من منع الإرسال المفرط دون المساس بالكود.

### 1.4 كيفية قراءة هذا الدليل

هذا الدليل مُصمَّم للمسح السريع، لا للقراءة الخطية. تظهر الجداول المرجعية في أعلى كل قسم رئيسي للبحث السريع. تتبع الأمثلة العملية الجداول المرجعية للقراء الذين يتعلمون بالأمثلة. يستخدم قسم استكشاف الأخطاء (15) تنسيق شجرة قرار قائم على العرَض — انتقل مباشرةً إلى عنوان العرَض، ثم اتبع فحوصات التحقق بالترتيب. لأي نوع تنبيه محدد، انتقل إلى القسم 8 وابحث عن H3 المقابل؛ كل إدخال مستقل بذاته ويتضمن المُحفِّز والحد والمستلمين والعناصر النائبة ومثالاً عمليًا والأمر الدقيق للاختبار.

## 2. البنية في لمحة {#2-architecture-at-a-glance}

### 2.1 خط الأنابيب ذو المراحل الخمس

يمر إرسال كل تنبيه بالمراحل الخمس التالية، سواء كان التنبيه مستندًا إلى حدث أو مجدولًا:

```
┌───────────────┐     ┌─────────────┐     ┌──────────────┐     ┌───────────┐     ┌──────────┐
│   1. Trigger  │────►│ 2. Rule     │────►│ 3. Data      │────►│ 4. Queue  │────►│ 5. Deliv-│
│               │     │    Match    │     │    Query     │     │           │     │    ery   │
│ afterSave /   │     │             │     │              │     │           │     │          │
│ cron job      │     │ alert_rules │     │ getPending   │     │ alert_    │     │ Email /  │
│               │     │ lookup      │     │ Items()      │     │ queue     │     │ SMS      │
└───────────────┘     └─────────────┘     └──────────────┘     └───────────┘     └──────────┘
```

**المرحلة 1 — المُحفِّز:** يقع حدث مجالي (حفظ سجل في CakePHP) أو يستدعي المُجدوِل `alerts:check` في دورة زمنية.

**المرحلة 2 — مطابقة القاعدة:** يقوم النظام بتحميل كل سجلات `alert_rules` التي يطابق حقل `feature` فيها نوع التنبيه المُحفَّز. يتم تخطي القواعد المعلَّمة `Enabled = No`.

**المرحلة 3 — استعلام البيانات:** يستدعي أمر artisan الدالة `getPendingItems()` لجلب السجلات التي تحقق شرط الحد. لكل سجل، تبني `fillPlaceholders()` رسالة مُخصَّصة من القالب.

**المرحلة 4 — قائمة الانتظار:** يستدعي الأمر `processContactList()` لكل مستلِم يتم تحديده. يتم حساب بصمة SHA-256 للعنوان ومحتوى الرسالة. إذا كانت بصمة متطابقة موجودة في `alert_logs` بحالة `PENDING`، يتم تخطي العنصر بصمت. خلاف ذلك، يُدرَج صف في `alert_queue`.

**المرحلة 5 — الإرسال:** يقوم الأمر `alerts:send`، الذي يُشغَّل كل دقيقة بواسطة Laravel scheduler، بقراءة صفوف `alert_queue` ذات الحالة `PROCESSING` وإرسالها إلى خدمة البريد (SMTP) أو بوابة الرسائل القصيرة (Twilio). يتم تحديث الحالة إلى `SENT` أو `FAILED` عند الاكتمال.

### 2.2 التنبيهات المستندة إلى الحدث مقابل التنبيهات المجدولة

| الخاصية | مستند إلى الحدث | مجدول |
|----------|-----------------|--------|
| مصدر المُحفِّز | استدعاء CakePHP `afterSave()` على النموذج | أمر cron `alerts:check` |
| التكرار في الواجهة | `Once` (يُطلَق مع كل حدث مُحفِّز) | `Daily` أو `Weekly` أو `Monthly` |
| زمن الاستجابة | شبه فوري (ثوانٍ بعد الحفظ) | حتى دورة جدولة واحدة |
| التحكم بالبدء/الإيقاف في الواجهة | غير متاح — يُطلَق من أحداث البيانات | يُتحكَّم به عبر إعداد التكرار |
| أمثلة | غياب الطالب، القبول، التسجيل، تغيير الحالة | تحذير التقاعد، صلاحية الترخيص، تصعيد القضية |

### 2.3 مسارا الإرسال

| المسار | المُحفِّز | الأوامر |
|--------|---------|--------|
| **مستند إلى الحدث** | CakePHP model `afterSave` → `AlertLogsTable::triggerLaravelAlertFromCakePHP()` | `alerts:student-absence`, `alerts:student-admission`, `alerts:student-enrolment`, `alerts:student-status-change` |
| **مجدول** | `alerts:check` (cron أو يدوي) | `alerts:retirement-warning`, `alerts:staff-employment`, `alerts:staff-leave`, `alerts:staff-type`, `alerts:system-updates`, `alerts:case-escalation`, `alerts:license-validity`, `alerts:license-renewal`, `alerts:scholarship-application`, `alerts:scholarship-disbursement` |

### 2.4 جداول قاعدة البيانات المعنية

| الجدول | الدور في خط الأنابيب |
|--------|----------------------|
| `alerts` | صف واحد لكل نوع تنبيه؛ يخزّن التكرار وحالة التفعيل |
| `alert_rules` | صف واحد أو أكثر لكل نوع تنبيه؛ يخزّن الحد والأدوار والعنوان والرسالة |
| `alert_queue` | صفوف إرسال مؤقتة؛ الحالة `0` (معلَّق)، `1` (مُرسَل)، `-1` (فشل) |
| `alert_logs` | سجل تدقيق دائم؛ صف واحد لكل رسالة مُرسَلة مع بصمة SHA-256 |
| `alert_logs_roles` | جدول وصل يربط `alert_logs` بـ `security_roles` |
| `security_groups` | تجميعات مستخدمين قائمة على الأدوار لتحديد المستلمين |
| `security_group_users` | عضوية المستخدمين في المجموعات الأمنية |
| `security_users` | سجلات جهات الاتصال؛ حقلا `email` و`mobile_number` يقودان الإرسال |
| `system_processes` | صف واحد لكل تشغيل أمر artisan؛ يخزّن الحالة ومسار ملف السجل |

## 4. إدارة جداول التنبيهات {#4-managing-alert-schedules}

### 4.1 قائمة أنواع التنبيهات

**لقطة شاشة 4.1** — قائمة التنبيهات مع عمود التكرار ومؤشرات الحالة.

![Alerts list](screenshots/alerts-list.png)

تعرض شاشة Alerts كل نوع تنبيه مسجَّل في النظام. يعرض كل صف اسم التنبيه والتكرار الحالي وحالة التشغيل. صفوف التنبيهات المستندة إلى الحدث تعرض التكرار `Once` ولا تحتوي على عناصر تحكم Start/Stop، لأنها تُطلَق تلقائيًا عند تغيّر البيانات وليس وفق جدول زمني.

### 4.2 خيارات التكرار

| الخيار | السلوك |
|--------|--------|
| `Never` | معطَّل تمامًا؛ لا يتم إطلاق أي تنبيه بصرف النظر عن تكوين القاعدة |
| `Once` | يُطلَق مرة واحدة لكل حدث مُحفِّز؛ مستخدم حصرًا للتنبيهات المستندة إلى الحدث |
| `Daily` | يعمل مرة واحدة على الأكثر لكل يوم تقويمي لكل سجل مطابق |
| `Weekly` | يعمل مرة واحدة على الأكثر لكل فترة 7 أيام لكل سجل مطابق |
| `Monthly` | يعمل مرة واحدة على الأكثر لكل شهر تقويمي لكل سجل مطابق |

> **ملاحظة:** `SystemUpdates` هو نوع التنبيه الوحيد الذي يأتي بـ `Daily` كتكراره الافتراضي. جميع أنواع التنبيهات الأخرى تأتي افتراضيًا بـ `Never`.

### 4.3 بدء وإيقاف تنبيه

لتفعيل تنبيه مجدول:

1. انتقل إلى **Administration → Communications → Alerts**.
2. اضغط **View** على نوع التنبيه الذي تريد تفعيله.
3. تحقق من أن التكرار مضبوط على `Daily` أو `Weekly` أو `Monthly`.
4. اضغط **Start** في شريط الإجراءات.

لتعطيل التنبيه، افتح السجل نفسه واضغط **Stop**. يعود نوع التنبيه إلى حالة متوقفة لكنه يحتفظ بإعداد التكرار الخاص به. لا يتم حذف أي قواعد موجودة.

> **ملاحظة:** أنواع التنبيهات المستندة إلى الحدث (غياب الطالب، قبول الطالب، تسجيل الطالب، تغيير حالة الطالب) لا تعرض عناصر تحكم Start/Stop. يتم إطلاقها تلقائيًا عند حفظ سجلاتها المُحفِّزة. لتعطيلها، اضبط التكرار على `Never`.

### 4.4 لماذا "Never" هو الإعداد الافتراضي الآمن

جميع أنواع التنبيهات باستثناء `SystemUpdates` تأتي افتراضيًا بـ `Never`. يمنع ذلك الإرسال العرضي على التثبيتات الجديدة أو بعد عمليات التهجير التي قد لا تكون القواعد فيها قد أُعدّت بعد. قبل ضبط أي نوع تنبيه على `Daily` أو `Weekly`، تأكد من وجود قاعدة مُفعَّلة واحدة على الأقل له وأن الحد وأدوار المستلمين مُكوَّنة بشكل صحيح.

## 5. قواعد التنبيهات — تكوين ما يتم إرساله {#5-alert-rules--configuring-what-to-send}

### 5.1 تشريح القاعدة

**لقطة شاشة 5.1** — قائمة قواعد التنبيهات تعرض جميع القواعد المُعدَّة.

![Alert Rules List](screenshots/alert-rules-list.png)

يحتوي كل سجل قاعدة تنبيه على الحقول التالية:

| الحقل | النوع | الغرض |
|-------|-------|-------|
| **Name** | نصي | تسمية وصفية للقاعدة؛ اجعلها فريدة عند تشارك عدة قواعد في الميزة نفسها |
| **Enabled** | نعم/لا | لا يتم تنفيذ القواعد المعطَّلة مطلقًا، بصرف النظر عن تكرار التنبيه |
| **Feature** | قائمة اختيار | نوع التنبيه الذي تحكمه هذه القاعدة؛ يجب أن يطابق مفتاح الميزة بالضبط (انظر القسم 8) |
| **Method** | قائمة اختيار | `Email` أو `SMS` أو كلاهما |
| **Threshold** | نص/JSON | الشرط الذي يحدد السجلات المؤهَّلة؛ يختلف التنسيق حسب نوع التنبيه (انظر القسم 7) |
| **Security Roles** | اختيار متعدد | الأدوار التي يتلقى أعضاؤها هذا الإشعار |
| **Subject** | نصي | سطر عنوان الإشعار؛ يدعم رموز `${placeholder}` |
| **Message** | نصي | نص الإشعار؛ يدعم رموز `${placeholder}` |

### 5.2 إنشاء قاعدة

**لقطة شاشة 5.2** — نموذج Add Alert Rule مع جميع الحقول مرئية.

![Alert Rule Add](screenshots/alert-rule-add.png)

1. انتقل إلى **Administration → Communications → Alert Rules**.
2. اضغط **Add** في شريط الإجراءات.
3. أدخل **Name** وصفيًا للقاعدة.
4. اضبط **Enabled** على `Yes`.
5. اختر **Feature** (نوع التنبيه) من القائمة المنسدلة.
6. اختر **Method**: `Email` أو `SMS` أو كلاهما.
7. أدخل **Threshold** بالتنسيق المطلوب للميزة المختارة (انظر القسم 7 والقسم 8).
8. اختر واحدًا أو أكثر من **Security Roles** الذين سيتلقى أعضاؤهم الإشعار.
9. أدخل قالب **Subject**، باستخدام رموز `${placeholder}` عند الحاجة.
10. أدخل قالب **Message**، باستخدام رموز `${placeholder}` عند الحاجة.
11. اضغط **Save**.

> **مهم:** جميع الحقول التالية مطلوبة: Method، Security Roles، Threshold (لأنواع التنبيهات التي تستخدمه)، Subject، وMessage. سيؤدي حفظ قاعدة بحقول مطلوبة ناقصة إلى ظهور خطأ تحقق.

### 5.3 تحرير وحذف قاعدة

**لقطة شاشة 5.3** — نموذج Edit Alert Rule يعرض قاعدة تصعيد قضية.

![Alert Rule Edit](screenshots/alert-rule-edit.png)

لتحرير قاعدة موجودة، انتقل إلى **Alert Rules**، اضغط **View** على القاعدة المستهدَفة، ثم اضغط **Edit** في شريط الإجراءات. عدِّل أي حقل واضغط **Save**.

لحذف قاعدة، افتح سجل القاعدة واضغط **Delete** في شريط الإجراءات. الحذف دائم. يتم الاحتفاظ بسجلات `alert_logs` المرتبطة لأغراض التدقيق.

> **مهم:** حذف القاعدة الوحيدة لنوع تنبيه يُعطِّل ذلك النوع فعليًا. لا يشير تأكيد الحذف إلى إدخالات السجل المرتبطة — تأكد من أن القاعدة لم تعد مطلوبة قبل الحذف.

### 5.4 قواعد متعددة لكل نوع تنبيه

هذا هو أهم مفهوم تكوين في وحدة التنبيهات. يمكن أن يحتوي نوع تنبيه واحد على أي عدد من القواعد، لكل منها حد مختلف وأدوار مستلمين مختلفة ورسالة مختلفة. تُنفَّذ القواعد باستقلال — قد يطابق سجل واحد عدة قواعد في اليوم نفسه.

**مثال 1 — صلاحية الترخيص: استراتيجية تصعيد من أربع طبقات**

| اسم القاعدة | الحد | التكرار | الأدوار | الغرض |
|-------------|------|---------|---------|-------|
| License Validity — 60 Days | `{"value": 60, "license_type": 3, "condition": 1}` | Weekly | HR Officer | أفق تخطيط مبكر |
| License Validity — 30 Days | `{"value": 30, "license_type": 3, "condition": 1}` | Daily | HR Officer, Principal | مرحلة تذكير نشطة |
| License Validity — 7 Days | `{"value": 7, "license_type": 3, "condition": 1}` | Daily | HR Officer, Principal, District HR | تصعيد عاجل قبل الانتهاء |
| License Validity — Expired | `{"value": 7, "license_type": 3, "condition": 2}` | Daily | HR Officer, Principal, District HR, Ministry | تنبيه امتثال بعد الانتهاء |

موظف تنتهي صلاحية ترخيصه بعد 5 أيام يُحفِّز القواعد الثلاث السابقة للانتهاء في وقت واحد. بمجرد انتهاء الترخيص، تُفعَّل القاعدة التالية للانتهاء. تشترك القواعد الأربع في مفتاح الميزة `LicenseValidity` نفسه.

**مثال 2 — تصعيد القضية: إشعار إداري طبقي**

| اسم القاعدة | الحد | الأدوار | الغرض |
|-------------|------|---------|-------|
| Case Escalation — 3 Days | `{"value": 3, "workflow_steps": [12]}` | Principal | تنبيه فوري لمالك القضية |
| Case Escalation — 7 Days | `{"value": 7, "workflow_steps": [12]}` | Principal, Coordinator | مسار تصعيد قياسي |
| Case Escalation — 21 Days | `{"value": 21, "workflow_steps": [12]}` | Principal, District Officer, Ministry | تصعيد حرج إلى الإدارة العليا |

قضية عمرها 25 يومًا تُحفِّز القواعد الثلاث في اليوم نفسه وتستمر في ذلك يوميًا حتى تتغير حالة القضية.

### 5.5 شروط القاعدة التي توقف التنفيذ

لا تُنفَّذ القاعدة إذا انطبق أي من الشروط التالية:

| الشرط | التأثير |
|-------|---------|
| `Enabled = No` على القاعدة | يتم تخطي القاعدة كليًا أثناء المعالجة |
| تكرار نوع التنبيه مضبوط على `Never` | لا تُنفَّذ أي قاعدة لذلك النوع |
| JSON الحد غير صالح أو غير قابل للتحليل | يتم تخطي القاعدة بصمت؛ افحص السجلات |
| لا توجد أدوار أمنية مُعيَّنة | لا يمكن تحديد مستلمين؛ لا تُوضَع رسائل في قائمة الانتظار |
| لا توجد سجلات `security_users` تطابق الأدوار المُعيَّنة | لا تُوضَع رسائل في قائمة الانتظار؛ هذا ليس خطأ |

## 6. العناصر النائبة {#6-العناصر-النائبة}

### 6.1 بناء جملة العنصر النائب

تستخدم العناصر النائبة الصيغة `${entity.field}`. أنها حساسة لحالة الأحرف. يتم استبدال العنصر النائب في وقت إنشاء الرسالة بالقيمة المقابلة من سجل البيانات المحل. إذا لم يتم التعرف على رمز العنصر النائب أو كانت قيمة الحقل فارغة، يتم ترك الرمز كنص حرفي في الإخراج — لا يتم إزالته بصمت.

تظهر رموز العناصر النائبة نفسها بشكل متطابق في قوالب الموضوع والرسالة.

### 6.9 السلوك عند تكون العنصر النائب فارغًا أو غير موجود

| الحالة | النتيجة |
|-----------|--------|
| الرمز موجود، قيمة الحقل فارغة | يبقى الرمز كنص حرفي في الإخراج (مثل `${staff.middle_name}` إذا كان الاسم الأوسط فارغًا) |
| الرمز مُخطأ في الكتابة (غير معروف) | يبقى الرمز كنص حرفي — لا يتم رفع أي خطأ |
| الرمز موجود، قيمة الحقل سلسلة فارغة | يتم استبدال سلسلة فارغة (يختفي الرمز من الإخراج) |
| الرمز مستخدم في الموضوع، الحقل غير متاح لنوع التنبيه | يبقى الرمز كنص حرفي |

> **مهم:** اختبر دائماً قوالب الرسائل مع السجلات الفعلية في بيئة تطوير قبل نشرها في الإنتاج. لا تمنع الرموز غير المحللة في أسطر الموضوع الإرسال — فهي تظهر كنص حرفي `${...}` في صندوق الوارد للمستقبِل.

## 7. الحدود الدنيا {#7-الحدود-الدنيا}

### 7.1 نظرة عامة على تنسيقات الحدود

| نوع التنبيه | مفتاح الميزة | التنسيق | مثال |
|-------|-------------|--------|---------|
| غياب الطالب | `StudentAttendance` | عدد صحيح | `5` |
| قبول الطالب | `StudentAdmission` | لا يوجد | _(غير مستخدم)_ |
| تسجيل الطالب | `StudentEnrolment` | لا يوجد | _(غير مستخدم)_ |
| تغيير حالة الطالب | `StudentStatus` | JSON (انتقال الحالة) | `{"old_status_id": 1, "new_status_id": 5}` |
| تحذير التقاعد | `RetirementWarning` | JSON (نافذة أيام) | `{"value": 90}` |
| انتهاء التوظيف | `StaffEmployment` | JSON (نافذة أيام) | `{"value": 30}` |
| انتهاء الإجازة | `StaffLeave` | JSON (أيام + نوع الإجازة) | `{"value": 3, "staff_leave_type": 2}` |
| نوع الموظف | `StaffType` | JSON (أيام + نوع الموظف) | `{"value": 30, "staff_type_id": 2}` |
| صحة الترخيص | `LicenseValidity` | JSON (أيام + نوع + حالة) | `{"value": 30, "license_type": 3, "condition": 1}` |
| تجديد الترخيص | `LicenseRenewal` | JSON (أيام + نوع + حالة + CPD) | `{"value": 60, "license_type": 3, "condition": 1, "training_categories": [1,2], "hour": 20}` |
| طلب المنحة الدراسية | `ScholarshipApplication` | JSON (أيام + حالة + فئة) | `{"value": 7, "condition": 1, "category": "PENDING"}` |
| صرف المنحة الدراسية | `ScholarshipDisbursement` | JSON (أيام + حالة) | `{"value": 7, "condition": 1}` |
| تصعيد القضية | `CaseEscalation` | JSON (أيام + خطوات سير العمل) | `{"value": 7, "workflow_steps": [12]}` |
| تحديثات النظام | `SystemUpdates` | لا يوجد | _(غير مستخدم)_ |

### 7.2 حقل `value`

حقل `value` هو عدد صحيح موجب يعبر عن عدد الأيام. معناه محدد لكل نوع تنبيه:

| نوع التنبيه | معنى `value` |
|-------|--------------|
| `RetirementWarning` | أيام قبل تاريخ التقاعد المحسوب |
| `StaffEmployment` | أيام قبل تاريخ انتهاء عقد التوظيف |
| `StaffLeave` | أيام قبل تاريخ انتهاء الإجازة الموافقة عليها |
| `StaffType` | أيام قبل مراجعة العقد أو تاريخ الانتهاء |
| `LicenseValidity` | أيام قبل (حالة 1) أو بعد (حالة 2) انتهاء صلاحية الترخيص |
| `LicenseRenewal` | أيام قبل انتهاء صلاحية الترخيص لبدء فحص ساعات CPD |
| `ScholarshipApplication` | أيام قبل تاريخ إغلاق تطبيق المنحة الدراسية |
| `ScholarshipDisbursement` | أيام قبل (حالة 1) أو بعد (حالة 2) تاريخ الصرف |
| `CaseEscalation` | عمر القضية بالأيام؛ تُصعّد القضايا التي تتجاوز هذه القيمة |

قيمة `value` بقيمة `0` أو رقم سالب لا معنى له لأي نوع تنبيه. استخدم عدداً صحيحاً موجباً.

### 7.3 حقل `condition`

يتحكم حقل `condition` في ما إذا كان التنبيه ينطلق للسجلات التي تقترب من تاريخ (قبل) أو السجلات التي تجاوزته بالفعل (بعد).

| القيمة | الاتجاه | نمط SQL |
|-------|-----------|-------------|
| `1` | قبل التاريخ — التاريخ في المستقبل، ضمن `value` يوماً | `DATEDIFF(target_date, NOW()) BETWEEN 0 AND value` |
| `2` | بعد التاريخ — التاريخ قد مضى، ضمن `value` يوماً | `DATEDIFF(NOW(), target_date) BETWEEN 0 AND value` |

يُستخدم من قبل: `LicenseValidity` و `LicenseRenewal` و `ScholarshipApplication` و `ScholarshipDisbursement`.

### 7.4 حدود خطوات سير العمل

يتم استخدام حقل `workflow_steps`، الذي تستخدمه `CaseEscalation`، مصفوفة JSON من معرّفات الأعداد الصحيحة من جدول `workflow_steps`. معرّفات الخطوات خاصة بالنشر ويجب الاستعلام عنها من قاعدة البيانات قبل إعداد أي قاعدة.

```sql
SELECT ws.id, ws.name, w.name AS workflow
FROM workflow_steps ws
JOIN workflows w ON w.id = ws.workflow_id
ORDER BY w.name, ws.name;
```

يمكن تحديد معرّفات خطوات متعددة. القضية التي يطابق `status_id` أي معرّف في المصفوفة مؤهلة للتصعيد.

```json
{"value": 7, "workflow_steps": [12, 13]}
```

### 7.5 فلاتر الفئة والنوع

عدة أنواع تنبيهات تدعم حقول فلاتر اختيارية أو مطلوبة تقيد المطابقة إلى أنواع سجل محددة:

| الحقل | نوع التنبيه | السلوك عند الحذف | الاستعلام للعثور على معرّفات صحيحة |
|-------|-------|----------------------|------------------------|
| `staff_leave_type` | `StaffLeave` | جميع أنواع الإجازات متطابقة | `SELECT id, name FROM staff_leave_types ORDER BY name;` |
| `staff_type_id` | `StaffType` | مطلوب — الحذف يسبب عدم مطابقة أي سجلات | `SELECT id, name FROM staff_types ORDER BY name;` |
| `license_type` | `LicenseValidity`, `LicenseRenewal` | مطلوب — بدونه، يتم مسح جميع أنواع الترخيص بشكل متزامن | `SELECT id, name FROM license_types ORDER BY name;` |
| `training_categories` | `LicenseRenewal` | مصفوفة مطلوبة — فقط التدريبات في الفئات المدرجة تحسب من `hour` | `SELECT id, name FROM staff_training_categories ORDER BY name;` |
| `category` | `ScholarshipApplication` | مطلوب — التصفية إلى فئة خطوة سير العمل (مثل `"PENDING"`) | `SELECT DISTINCT category FROM workflow_steps WHERE category IS NOT NULL;` |

### 7.6 إنشاء قواعد مزدوجة قبل / بعد

لأنواع التنبيهات التي تدعم `condition`، أنشئ قاعدتين منفصلتين لتغطية نافذتي الاقتراب والتأخر:

```json
// القاعدة أ — تحذير مسبق (حالة 1)
{
  "value": 30,
  "license_type": 3,
  "condition": 1
}
```

```json
// القاعدة ب — متابعة الامتثال بعد انتهاء الصلاحية (حالة 2)
{
  "value": 7,
  "license_type": 3,
  "condition": 2
}
```

كلا القاعدتين تشير إلى مفتاح الميزة نفسه. يتم تنفيذها بشكل مستقل. تنطلق القاعدة أ بينما الترخيص لا يزال صحيحاً لكن يقترب من انتهاء الصلاحية. تنطلق القاعدة ب بعد انتهاء صلاحية الترخيص. يمكن تعيين موضوعات ورسائل وأدوار متلقي مختلفة على كل منها، مما يتيح نبرات اتصالات مختلفة للحالات القادمة مقابل المتأخرة.

> ```

**ملاحظات:** يتم تجميع جميع الغيابات للطالب في الفترة الأكاديمية المعطاة في عنصر قائمة انتظار واحد. الأمر لا ينطلق لكل سجل غياب فردي.

## 9. التنبيهات المُطلقة بسير العمل {#9-التنبيهات-المطلقة-بسير-العمل}

### 9.1 كيفية اختلاف تنبيهات سير العمل عن التنبيهات المستندة إلى القواعد

التنبيهات المستندة إلى القواعد (الأقسام 4-8) هي **قاعدة تنبيه موجودة بشكل دائم** مخزنة في جدول `alert_rules`. تنطلق بناءً على محفزات مجدولة أو قائمة على الحدث.

تنبيهات سير العمل هي **تنبيهات معرفة مؤقتاً** المرتبطة بحالة انتقال واحدة معينة. عندما ينتقل كائن إلى تلك الحالة، يمكن إرسال تنبيه **إذا تم تعيين قالب الموضوع والرسالة** في خطوة سير العمل نفسها.

**المميزات:**
- **ملموضة:** كل حالة انتقال لها التنبيه الخاص بها.
- **تفاعلية:** يتم عزل الإشعار بخطوة العمل، لا يتم دمجه مع التنبيهات الأخرى.
- **نطاق محدود:** لا توجد جداول أم `alert_rules` أو `alert_queue` — التنبيه ينطلق مباشرة.

### 9.2 شروط تثبيط تنبيه سير العمل

تنبيهات سير العمل **محظورة تماماً إذا كان أي من هذه صحيح:**

| الحالة | السلوك |
|--------|--------|
| خطوة سير العمل ليس لديها Subject و Message معرفة | لا يتم إرسال تنبيه |
| المتلقي (الدور الأمني) ليس لديه بريد إلكتروني و لا رقم هاتف | الرسالة تخزّن في `alert_queue` بحالة `pending` ، ولا تُرسل أبداً |
| `institution_configuration` تعطل الميزة (مثل `DISABLE_WORKFLOW_ALERTS = 1`) | تنبيهات سير العمل معطلة تماماً |

### 9.3 إعداد تنبيه سير العمل

1. انتقل إلى **Workflow Editor** في CakePHP.
2. اختر سير العمل والخطوة.
3. حدد **Subject** و **Message** - استخدم الرموز `${placeholder}` بنفس الطريقة كما في `alert_rules`.
4. انقر **Save**.
5. من الآن فصاعداً، سيتم إرسال تنبيه عندما ينتقل أي كائن إلى تلك الخطوة.

## 10. قائمة انتظار التنبيهات — خط أنابيب الإرسال {#10-قائمة-انتظار-التنبيهات--خط-أنابيب-الإرسال}

### 10.1 الغرض من شاشة قائمة الانتظار

جدول `alert_queue` يسجل كل **محاولة إرسال** في الانتظار أو المرسلة أو الفاشلة. كل صف يمثل إرسالاً واحداً إلى متلقٍ واحد.

شاشة **Alert Queue** تعرض جميع العناصر المعلقة والمرسلة والفاشلة. المسؤولون يستخدمونها للتحقق من الحالة، وحذف الرسائل المرفوضة، وإعادة محاولة الإرسالات الفاشلة يدويًا.

### 10.2 أعمدة قائمة الانتظار

| العمود | النوع | الوصف |
|--------|--------|-----------|
| `id` | Integer (PK) | معرّف الصف الفريد |
| `institution_id` | Integer (FK) | المؤسسة المالكة |
| `feature` | String | مفتاح الميزة (مثل `StudentAttendance`) |
| `rule_id` | Integer (FK) | معرّف `alert_rules` (null للتنبيهات غير المستندة إلى القواعد) |
| `status` | Enum | `pending` أو `sent` أو `failed` |
| `recipient_email` | VARCHAR(255) | بريد إلكتروني المتلقي |
| `recipient_phone` | VARCHAR(20) | رقم هاتف المتلقي |
| `subject` | TEXT | موضوع الرسالة المحلل |
| `message` | LONGTEXT | نص الرسالة المحلل |
| `checksum` | CHAR(64) | بصمة SHA-256 للموضوع + الرسالة + البريد الإلكتروني |
| `attempts` | TINYINT | عدد محاولات الإرسال |
| `last_attempt` | TIMESTAMP | آخر محاولة إرسال |
| `error_message` | TEXT | رسالة الخطأ الأخيرة (إن وجدت) |
| `created_at` | TIMESTAMP | عند الإنشاء |
| `updated_at` | TIMESTAMP | آخر تحديث |

### 10.3 رموز الحالة

| الحالة | المعنى | الإجراء المتاح |
|--------|--------|---------|
| `pending` | جاهز للإرسال ؛ بانتظار معالج الطابور | حذف، أو انتظر معالج الطابور |
| `sent` | أرسل بنجاح | حذف فقط |
| `failed` | فشل الإرسال بعد محاولات متعددة | حذف، أو اختياراً أعد محاولة يدويًا |

### 10.4 حذف عناصر قائمة الانتظار بكميات كبيرة

**لا تحذف مباشرة من الجدول.** استخدم واجهة المستخدم:

1. انتقل إلى **Administration → Communications → Alert Queue**.
2. حدد الصفوف التي تريد حذفها باستخدام خانات الاختيار.
3. انقر **Delete Selected**.
4. تأكد الحذف.

### 10.5 مخطط دورة حياة قائمة الانتظار

```
القاعدة ينطلق → عنصر جديد (status = pending)
                     ↓
        معالج الطابور يحاول الإرسال
                     ↓
         ✓ نجح → (status = sent) → محفوظ
                     ↗
         ✗ فشل → (status = failed) → محفوظ مع رسالة الخطأ
```

## 11. سجلات التنبيهات — سجل التدقيق {#11-سجلات-التنبيهات--سجل-التدقيق}

### 11.1 الغرض من سجلات التنبيهات

جدول `alert_logs` يسجل كل **الإرسالات الناجحة بشكل دائم** — المرسلة والمعالجة والسجل. على عكس `alert_queue` (وقت التشغيل المؤقت)، `alert_logs` دائم للتدقيق والامتثال.

يعرض السجل:
- من تم إرسال التنبيه إليه
- متى
- ماذا كان الموضوع والرسالة
- هل كان تكراراً ؛ (checksum يطابق رسالة سابقة)

### 11.2 إزالة التكرار عبر بصمات SHA-256

قبل إدراج صف جديد في `alert_logs`، يحسب النظام بصمة SHA-256 لـ:

```
SHA256(subject + "|" + message + "|" + recipient_email)
```

إذا وجدت نفس البصمة في آخر 7 أيام، **يتم تخطي الإرسال ويتم وضع علامة كـ `deduplicated`**.

**الفائدة:** لا تسيل متلقٍ برسائل متطابقة. قواعد متعددة قد تحسب الذات نفسها — يتم الكشف عنها.

### 11.3 عرض وحذف إدخالات السجل الفردية

**عرض:**
1. انتقل إلى **Administration → Communications → Alert Logs**.
2. ابحث أو تصفية حسب التاريخ أو الميزة أو المتلقي.
3. انقر على صف لعرض الموضوع والرسالة الكاملة.

**حذف:**
1. انقر على **Delete** بجانب الإدخال.
2. تأكد الحذف.

### 11.4 حذف إدخالات السجل بكميات كبيرة

**لا تحذف مباشرة.** استخدم واجهة المستخدم:

1. في **Alert Logs**، حدد الإدخالات باستخدام خانات الاختيار.
2. انقر **Delete Selected**.
3. تأكد.

> **تحذير:** حذف السجلات يقلل من سجل التدقيق. احفظ نسخة احتياطية قبل الحذف بكميات كبيرة.

## 13. التكوين التشغيلي {#13-التكوين-التشغيلي}

### 13.1 مفاتيح الإعدادات

تكوينات التنبيهات مخزنة في جدول `configuration` تحت الفئة `alerts`.

| المفتاح | المعنى | القيمة الافتراضية |
|--------|--------|---------|
| `ALERTS_ENABLED` | تفعيل / تعطيل جميع التنبيهات | `1` |
| `ALERTS_PROCESS_LIMIT` | الحد الأقصى لعدد عناصر الطابور المعالجة لكل تشغيل | `100` |
| `ALERTS_WORKING_HOURS_FROM` | ساعة بدء العمل (24 ساعة) | `08` |
| `ALERTS_WORKING_HOURS_TO` | ساعة إنهاء العمل (24 ساعة) | `18` |
| `ALERTS_RESPECT_WEEKENDS` | تخطي نهايات الأسبوع | `1` |
| `ALERTS_LOCAL_TIMEZONE` | المنطقة الزمنية للمؤسسة | `UTC` |

### 13.2 تقليل التدفق عبر ALERTS_PROCESS_LIMIT

عند تشغيل `alerts:send-queue`، يتم معالجة **حد أقصى من عناصر الطابور** لكل تشغيل. السبب:

- **الأداء:** معالجة الآلاف من الرسائل مرة واحدة قد تثقل النظام.
- **الاستقرار:** تقسيم العمل يمنع انقطاع الاتصال.

**التعديل:**
```sql
UPDATE configuration SET value = '500' 
WHERE key = 'ALERTS_PROCESS_LIMIT' 
  AND category = 'alerts';
```

ثم امسح الكاش:
```bash
cd /var/www/html/emis/core/api && php artisan config:cache
```

### 13.3 تقييد الإرسال إلى ساعات العمل

تعطيل التنبيهات خارج ساعات العمل:

```sql
UPDATE configuration SET value = '1' 
WHERE key = 'ALERTS_RESTRICT_TO_WORKING_HOURS' 
  AND category = 'alerts';
```

الآن، أي تنبيه ينطلق خارج الساعات المحددة في `ALERTS_WORKING_HOURS_FROM` و `ALERTS_WORKING_HOURS_TO` سيؤجل إلى الصباح.

### 13.4 احترام نهايات الأسبوع المحلية

بعض المناطق لها نهايات أسبوع مختلفة (الجمعة / السبت بدلاً من السبت / الأحد). قم بتعديل:

```sql
UPDATE configuration SET value = 'fri,sat' 
WHERE key = 'ALERTS_WEEKEND_DAYS' 
  AND category = 'alerts';
```

الآن، التنبيهات المجدولة ستتخطى الجمعة والسبت.

### 13.5 إعداد Cron

أضف سطراً إلى `crontab` المضيف:

```bash
* * * * * cd /var/www/html/emis/core/api && php artisan alerts:check >> /var/log/openemis-alerts.log 2>&1
* * * * * cd /var/www/html/emis/core/api && php artisan alerts:send-queue >> /var/log/openemis-alerts.log 2>&1
```

**كل دقيقة:**
1. `alerts:check` — يفحص القواعس المجدولة ويضع البنود في الطابور.
2. `alerts:send-queue` — يرسل عناصر الطابور.

> **ملاحظة:** النظام **لا يتطلب** cron إذا كانت التنبيهات معتمدة على الحدث فقط (قاعدة واحدة في كل مرة). Cron مطلوب فقط للتنبيهات المجدولة.

## 14. إجراءات الاختبار والتشغيل الجاف {#14-إجراءات-الاختبار-والتشغيل-الجاف}

### 14.1 تشغيل أمر مباشر

اختبر قاعدة واحدة دون الانتظار لـ cron:

```bash
docker exec poe-application /bin/sh -c \
  "cd /var/www/html/emis/core/api && php artisan alerts:student-absence \
   --user_id=1 --rule_id=3 --process_id=0 --student_id=abc123 --academic_period_id=5"
```

**معاملات:**
- `--user_id=1` — المستخدم الذي يشغل الأمر
- `--rule_id=3` — معرّف القاعدة من `alert_rules`
- `--process_id=0` — معرّف `system_processes` (0 = بدون تتبع)
- `--student_id=abc123` — معرّف الطالب (اختلف حسب نوع التنبيه)

### 14.2 فحص قائمة الانتظار وعمليات النظام

**من داخل المتصفح:**
1. انتقل إلى **Administration → Communications → Alert Queue**.
2. تحقق من الحالات — يجب أن تكون `pending` أو `sent`.

**من CLI:**
```sql
SELECT COUNT(*) FROM alert_queue WHERE status = 'pending';
SELECT * FROM system_processes ORDER BY created DESC LIMIT 5;
```

### 14.3 ملء رسائل البريد الإلكتروني وأرقام الهواتف المفقودة (قواعد البيانات للتطوير / الاختبار فقط)

في **قاعدة بيانات التطوير فقط**:

```sql
UPDATE security_users 
SET email = CONCAT('user', id, '@example.com')
WHERE email IS NULL OR email = '';

UPDATE security_users 
SET telephone = CONCAT('+1-555-', LPAD(id, 4, '0'))
WHERE telephone IS NULL OR telephone = '';
```

هذا يضمن أن كل مستخدم لديه بريد إلكتروني وهاتف لأغراض الاختبار.

### 14.4 التحقق من أن قاعدة البيانات مجهولة الهوية

**تأكد من أن قاعدة البيانات الخاصة بك ليست الإنتاج:**

```bash
mysql -h 127.0.0.1 -u root -prootpassword openemis_core_v5 -e \
  "SELECT COUNT(*) as user_count FROM security_users; \
   SELECT COUNT(*) as institution_count FROM institutions;"
```

- **الإنتاج:** عشرات الآلاف من المستخدمين والمؤسسات
- **التطوير:** عشرات فقط

### 14.5 فرض تشغيل جميع التنبيهات المجدولة الآن

**تحذير:** سيرسل جميع التنبيهات المجدولة **فوراً**.

```bash
docker exec poe-application /bin/sh -c \
  "cd /var/www/html/emis/core/api && php artisan alerts:check --force"
```

## 15. استكشاف الأخطاء {#15-استكشاف-الأخطاء}

### 15.1 تم إطلاق التنبيه ولم يتم استقبال البريد الإلكتروني {#151-تم-إطلاق-التنبيه-ولم-يتم-استقبال-البريد-الإلكتروني}

**خطوات التشخيص:**

1. **تحقق من عنصر قائمة الانتظار:**
   ```sql
   SELECT * FROM alert_queue 
   WHERE recipient_email = 'user@example.com' 
   ORDER BY created_at DESC LIMIT 1;
   ```
   - الحالة `pending`؟ → معالج الطابور لم يعمل بعد
   - الحالة `failed`؟ → انظر `error_message`
   - الحالة `sent`؟ → انتقل إلى الخطوة 2

2. **تحقق من مسؤول البريد:**
   ```bash
   tail -50 /var/log/mail.log  # على المضيف
   ```

3. **تحقق من حساب البريد نفسه:**
   - هل عنوان البريد صحيح؟ تحقق من `security_users.email`
   - هل يتم حظره بواسطة جدار حماية؟

4. **أعد محاولة الإرسال يدويًا:**
   ```bash
   docker exec poe-application /bin/sh -c \
     "cd /var/www/html/emis/core/api && \
      php artisan alerts:send-queue --limit=1"
   ```

### 15.2 تم تفعيل قاعدة التنبيه ولكن لا تُطلق أبداً {#152-تم-تفعيل-قاعدة-التنبيه-ولكن-لا-تُطلق-أبدا}

**خطوات التشخيص:**

1. **تحقق من أن القاعدة مفعلة:**
   ```sql
   SELECT id, name, enabled FROM alert_rules 
   WHERE name LIKE '%your rule%';
   ```
   - `enabled = 0`؟ → تفعيلها

2. **تحقق من أن الحدث ينطلق:**
   - لتنبيهات مستندة إلى الحدث، تحقق من أن الحدث **قد حدث بالفعل** (غياب، قبول، إلخ)
   - شغل الأمر اليدوي من §14.1

3. **تحقق من التصفية:**
   - هل `threshold` صحيح؟
   - هل `security_roles` معيّنة؟

4. **تحقق من Cron:**
   ```bash
   grep CRON /var/log/syslog | tail -20
   ```

### 15.3 خطأ في التحقق من الصحة عند حفظ قاعدة {#153-خطأ-في-التحقق-من-الصحة-عند-حفظ-قاعدة}

**الخطأ الشائع:** "معرّف دور الأمان غير صحيح"

**الحل:** 
1. تأكد من أنك تحدد دوراً واحداً على الأقل في **Security Roles**.
2. أضف `threshold` إذا لزم الأمر لنوع التنبيه.
3. أضف `subject` و `message`.

### 15.4 رموز العناصر النائبة تظهر بشكل حرفي في الرسائل المُرسلة {#154-رموز-العناصر-النائبة-تظهر-بشكل-حرفي-في-الرسائل-المرسلة}

**السبب:** الرمز `${...}` غير معروف أو الحقل فارغ.

**الحل:**
1. تحقق من الإملاء: `${student.name}` ليس `${student.full_name}`
2. اختبر مع سجل حقيقي له قيمة في الحقل.
3. إذا كان يجب أن يكون الحقل فارغاً، أضف نص احتياطي: `"Name: ${student.name}, or Unknown if null"`

### 15.5 أعمدة قائمة الانتظار يُظهر "عنوان غير معروف" {#155-أعمدة-قائمة-الانتظار-يُظهر-عنوان-غير-معروف}

**السبب:** الرسالة لم تُحلل الرموز.

**الحل:**
1. تحقق من أن القاعدة لديها `subject` و `message`.
2. تحقق من أن معرّف الكائن المرتبط صحيح.
3. أعد تشغيل الأمر مع `--debug`.

### 15.6 الأداء بطيء عند معالجة آلاف عناصر قائمة الانتظار {#156-الأداء-بطيء-عند-معالجة-آلاف-عناصر-قائمة-الانتظار}

**الحل:**
1. قلل `ALERTS_PROCESS_LIMIT` إلى 50-100 (من 300).
2. قسّم معالجة الطابور إلى عدة Cron jobs:
   ```
   * * * * * ... php artisan alerts:send-queue --limit=50
   * * * * * ... php artisan alerts:send-queue --limit=50  # run again
   ```
3. أضف فهرساً على `alert_queue(status, created_at)`.

## 16. الملاحق {#16-الملاحق}

### 16.A قاموس بنية قاعدة البيانات {#16a-قاموس-بنية-قاعدة-البيانات}

الجداول الرئيسية المعنية بالتنبيهات:

**alert_rules**
- تخزين تعريفات التنبيهات الدائمة
- أعمدة: `id`, `institution_id`, `feature`, `enabled`, `method`, `threshold`, `subject`, `message`, `created_at`, `updated_at`

**alert_queue**
- الرسائل المعلقة والمرسلة
- أعمدة: `id`, `institution_id`, `feature`, `rule_id`, `status`, `recipient_email`, `recipient_phone`, `subject`, `message`, `checksum`, `attempts`, `last_attempt`, `error_message`

**alert_logs**
- سجل دائم للرسائل المرسلة
- أعمدة: `id`, `institution_id`, `feature`, `rule_id`, `recipient_email`, `subject`, `message`, `checksum`, `deduplicated`, `created_at`

### 16.B ملاحظات الحقول والحالات {#16b-ملاحظات-الحقول-والحالات}

| الحقل | النوع | ملاحظات |
|-------|--------|---------|
| `feature` | VARCHAR(255) | يجب أن يطابق مفتاح الميزة (مثل `StudentAttendance`) |
| `status` | ENUM('pending', 'sent', 'failed') | حالة الإرسال |
| `checksum` | CHAR(64) | SHA-256 لكشف التكرار |
| `enabled` | TINYINT(1) | 0 = معطل، 1 = مفعل |

### 16.C استعلامات SQL مرجعية {#16c-استعلامات-sql-مرجعية}

**عد التنبيهات المعلقة:**
```sql
SELECT COUNT(*) FROM alert_queue WHERE status = 'pending';
```

**عرض الأخطاء:**
```sql
SELECT recipient_email, error_message, COUNT(*) 
FROM alert_queue 
WHERE status = 'failed' 
GROUP BY recipient_email, error_message;
```

**تنظيف السجلات القديمة:**
```sql
DELETE FROM alert_logs WHERE created_at < DATE_SUB(NOW(), INTERVAL 90 DAY);
```

**البحث عن خطوات سير العمل لـ CaseEscalation:**
```sql
SELECT ws.id, ws.name, w.name AS workflow
FROM workflow_steps ws
JOIN workflows w ON w.id = ws.workflow_id
ORDER BY w.name, ws.name;
```

### 16.D قاموس المصطلحات {#16d-قاموس-المصطلحات}

| المصطلح | التعريف |
|--------|--------|
| **مفتاح الميزة** | معرّف نص دقيق لنوع التنبيه كما يظهر في عمود `alert_rules.feature`. حساس لحالة الأحرف. أمثلة: `StudentAttendance` و `CaseEscalation`. |
| **اسم العملية** | اسم فئة PHP المستخدمة داخلياً لتحديد أمر artisan وتعيينه إلى مساره. مثال: `AlertStudentAbsence`. مختلف عن اسم أمر artisan. |
| **بصمة** | بصمة SHA-256 للموضوع والنص المحلل للمتلقي. يستخدم للكشف والتكرار في `alert_logs.checksum`. |
| **محلل العناصر النائبة** | خدمة Laravel (`PlaceholderResolver`) مسؤولة عن استبدال رموز `${...}` بقيم البيانات الفعلية. |
| **حد JSON** | القيمة بصيغة JSON المخزنة في `alert_rules.threshold` التي تتحكم في السجلات المؤهلة للإشعار. تختلف التنسيقات والحقول المطلوبة حسب نوع التنبيه. |
| **عنصر الطابور** | صف واحد في `alert_queue` يمثل محاولة إرسال واحدة معلقة أو مرسلة أو فاشلة إلى متلقٍ واحد. |
| **عملية النظام** | صف في `system_processes` يمثل تنفيذ واحد لأمر artisan تنبيهات. يخزن اسم الأمر والحالة والمسار إلى ملف السجل. |
